优先考虑泛型方法

泛型方法的一个显著特征是：无需明确指定类型参数的值，不像调用泛型构造器的时候是必须指定的。
类型限制<T extends Comparable>可以读作：“针对可以与自身进行比较的每个类型T ”
泛型方法还有一个比较重要的概念叫做类型推导。
在创建一个泛型集合实例时，这样写 Map map = new HashMap();
我们在调用泛型HashMap的构造器时，在new HashMap之后使用指明了类型参数。那么我们在使用泛型方法的时候是否也需要这样做呢？
写出这样的代码 ：Set.union(s1,s2); 也就是说是否在需要调用方法的时候像调用构造器的那样在方法名后面指明类型参数。
泛型方法会根据具体传入的s1,s2的类型，来推断出类型参数E。如果传入的s1,s2是HashSet，那么类型参数E就是String；
如果传入的s1,s2是HashSet，那么类型参数E就是Integer。
一句话就是：泛型方法的类型参数根据传入的实际参数的类型而定，不需要再调用时显示指明。

泛型方法就像泛型一样，使用起来比要求客户端转换输入参数并返回值的方法来的更加安全，也更加容易。就像类型一样，你应该确保新方法可以不用转换就能使用，这通常意味着要将他们进行返型化，并且就像类型一样，还应该将现有的方法泛型化，不会破坏现有的客户端。

类和接口

模块设计好坏的评判标准:在于这个模块对于外部的其他模块而言是否隐藏其内部数据和其他实现细节.
高内聚,低耦合..低耦合是用来度量模块与模块直接的依赖关系,也就是说,模块之间的依赖关系越低越好.
信息隐藏的概念:一个模块不需要知道其他模块的内部工作情况,它可以有效地解除组成系统的各模块之间的耦合关系,加速系统的开发速度.并且在调试的时候不影响其他的模块,它本身并不会带来很好的性能,但是可以有效地调节性能.提高了软件的可重用性.降低了构建大型系统的风险.

java程序设计语言中提供了访问控制机制来决定类,接口和成员的可访问性.

访问修饰符:

  • 私有的(private):只有在声明该成员的顶层类内部才可以访问这个成员
  • 包级私有的(package-private):声明该成员的包内部的任何类都可以访问这个成员.被称为”缺省访问级别”;
  • 受保护的(“protected”):声明该成员的类的子类可以访问这个成员,并且声明该成员的包内部的任何类也可以访问这个成员.
  • 共有的(public):在任何地方都可以访问该成员
    注意：如果方法覆盖了超类中的一个方法,子类中的访问级别就不允许低于超类中的访问级别.

不能为了测试而将类,接口或者成员变成包的导出的API部分.
实力域不能是公有的,包含共有可变域的类并不是线程安全的.

长度非0的数组总是可变的,所以类具有公有的final数组域.或者返回这种域的访问方法,这几乎总是错误的.
总而言之,应始终尽可能地降低可访问性,在仔细地设计了一个最小的共有API之后应防止把任何散乱的类,接口和成员变成API的一部分,处了公有静态final域的特殊情形之外,公有类都不应该包含公有域,并且要确保共有静态final域所引用的对象都是不可变的.

请不要在新代码中使用原生态类型

泛型的概念：生命中具有一个或者多个类型参数的类或者接口，泛型类和接口统称为泛型。
每种泛型定义一种参数化的类型，构成格式为：List（字符串列表），表示元素类型为String的列表
每个泛型都定义了一个原生态类型，即不带任何实际类型参数的泛型名称。如：List

private final Collection stamps = …
通过这条声明，编译器知道stamps应该只包含Stamp实例，并给予保证，并且从集合中删除元素不需要进行手工转换。
如果使用原生态类型，就失掉了泛型在安全性和表述性方面的优势，但是还是允许使用原生态类型的。并且能够与使用泛型的新代码互用，才能将参数化类型的实例传递给那些被设计成普通类型的方法，这种需求被称作移植兼容性，促成了支持原生态类型的决定。

注意：可以将List传递给类型List的参数，但是不能将它传递给类型List

无限制的通配符类型：如果要使用泛型，但是不确定或者不关心实际的参数类型，就可以使用一个问号代替，符号表示：<?>
通配符类型和原生态类型的区别：通配符类型是安全的，但是原生态类型则不安全。由于可以将任何元素放进使用原生态类型的集合中，因此很容易破坏该集合的类型约束条件，但是不能将任何元素（null除外）放到Collection<?>中。

不要在新代码中使用原生态类型
两个例外：两者都源于“泛型信息可以在运行时被擦除”，
1.在类文字中必须使用原生态类型，规范不允许使用参数化类型，
2.与instanceof有关，泛型信息可以在运行时被擦除，在参数化类型而非无限制通配符的类型上使用instanceof是非法的。

考虑实现Comparable

compareTo是comparable接口中唯一的方法。
与equals方法不同的是，在跨越不同的类的时候，compareTo可以不做比较：如果两个被比较的对象引用不用类的对象，那么它可以抛出ClassCastException异常。
依赖于比较关系的类包括有序集合类TreeSet和TreeMap,以及工具类Collection和Arrays，他们内部都包含有搜索和排序算法。
告诫：无法在用新的值组建扩展可实例化的类时，同时保持compareTo约定，除非放弃面向对象的抽象优势。
另外就是compareTo方法中是不需要转换类型，因为它是进行排序的，而不是进行等同行的比较，所以不需要进行转换类型的，也不需要进行类型判断。
还有就是有多个关键域，那么就需要按照最关键的域开始进行比较，若相同，则依次向下比较，这样的结果才可能是希望看到的结果。

始终要覆盖toString

建议所有的类都覆盖这个方法。在实际应用中，toString方法应该返回对象中包含的所有值得关注的信息。
在实现toString方法的时候，必须做出一个很重要的决定：是否在文档中指定返回值的格式。如果指定了格式，最好再提供一个相匹配的静态工厂或者构造器，以便程序员可以很容易地在对象和它的字符串之间来回转换。
但是无论是否决定指定格式，都应该在编码中添加文档注释指明自己的意图。都应该提供一种编程式的访问途径

消除非受检警告

当泛型编程时，会遇到很多编译器警告：非受检强制转化警告、非受检方法调用警告、非受检普通数组创建警告、非受检转换警告
要尽可能消除每一个非受检警告。
如果无法消除警告，同时可以证明引起警告的代码是类型安全的，可以用一个@SuppressWarnings(“unchecked”)注解来禁止这条警告，可以用任何粒度的级别中，但是应该始终在尽可能小的范围中使用SuppressWarnings注解，永远不要在整个类中使用SuppressWarnings，可能会掩盖了重要的警告。
其次，将SuppressWarnings放在return语句中是非法的，应该声明一个局部变量来保存返回值，并注解其声明。

列表优先于数组

数组与泛型的区别一：数组是协变的，泛型是不可变的

数组与泛型的区别二：
数组是具体化的，数组只在运行时才知道并检查他的元素类型约束。
泛型在编译前就检查它的元素类型，在编译时强化它们的类型信息，在运行时丢弃(或者擦除)它的元素类型信息，而且泛型是通过擦除来实现的。

因此，数组和泛型不能很好地混合使用。
数组提供了运行时的类型安全，但是没有编译的类型安全，反之对于泛型也一样，如果发现将他们混合起来，应该用列表代替数组

优先考虑泛型

如果一不小心创建了泛型数组，在java中这是不允许的，但是有两种解决办法：一：直接绕过创建泛型数组的禁令：创建一个Object的数组，并将它转成泛型数组类型
Object object = (E[]) new Object[ARRAY_SIZE];
二：将elements域的类型从E[]改为Object[]。
使用泛型比使用需要在客户端代码中进行转换的类型来得更安全，也更容易，只要时间允许，就把现有的类型都返型化
