工厂方法模式

一、工厂模式定义

      工厂方法模式，又称工厂模式、多态工厂模式和虚拟构造器模式，通过定义工厂父类负责定义创建对象的公共接口，而子类则负责生成具体的对象。

二、主要作用

      将类的实例化（具体产品的创建）延迟到工厂类的子类（具体工厂）中完成，即由子类来决定应该实例化（创建）哪一个类。


三、解决的问题

工厂一旦需要生产新产品就需要修改工厂类的方法逻辑，违背了“开放 - 关闭原则

  1. 即简单工厂模式的缺点
  2. 之所以可以解决简单工厂的问题，是因为工厂方法模式把具体产品的创建推迟到工厂类的子类（具体工厂）中，此时工厂类不再负责所有产品的创建，而只是给出具体工厂必须实现的接口，这样工厂方法模式在添加新产品的时候就不修改工厂类逻辑而是添加新的工厂子类，符合开放封闭原则，克服了简单工厂模式中缺点


四、UML类图

五、使用步骤

  1. 创建抽象工厂类，定义具体工厂的公共接口
  2. 创建抽象产品类，定义具体产品的公共接口
  3. 创建具体产品类，继承抽象产品类，定义生产的具体产品
  4. 创建具体工厂类，继承抽象工厂类，定义创建对应具体产品实例的方法
  5. 外界通过调用具体工厂类的方法，从而创建不同具体产品类的实例


六、缺点

1.添加新产品时，除了增加新产品类外，还要提供与之对应的具体工厂类，系统类的个数将成对增加，在一定程度上增加了系统的复杂度；同时，有更多的类需要编译和运行，会给系统带来一些额外的开销；
2.由于考虑到系统的可扩展性，需要引入抽象层，在客户端代码中均使用抽象层进行定义，增加了系统的抽象性和理解难度，且在实现时可能需要用到DOM、反射等技术，增加了系统的实现难度。
3.虽然保证了工厂方法内的对修改关闭，但对于使用工厂方法的类，如果要更换另外一种产品，仍然需要修改实例化的具体工厂类；
4.一个具体工厂只能创建一种具体产品

七、应用场景

1.当一个类不知道它所需要的对象的类时
在工厂方法模式中，客户端不需要知道具体产品类的类名，只需要知道所对应的工厂即可；
2.当一个类希望通过其子类来指定创建对象时
在工厂方法模式中，对于抽象工厂类只需要提供一个创建产品的接口，而由其子类来确定具体要创建的对象，利用面向对象的多态性和里氏代换原则，在程序运行时，子类对象将覆盖父类对象，从而使得系统更容易扩展。
3.将创建对象的任务委托给多个工厂子类中的某一个，客户端在使用时可以无须关心是哪一个工厂子类创建产品子类，需要时再动态指定，可将具体工厂类的类名存储在配置文件或数据库中。

基本类型优于装箱基本类型

什么是装箱类型？

      java中包含8种基本数据类型和3中引用数据类型,其中每个基本数据类型都有一个对应的引用类型,称作装箱基本类型,装箱基本数据类型中对应于int,double和boolean的是Integer,Double和Boolean

java 1.5之后增加了自动装箱和自动拆箱.

在基本数据类型和装箱基本类型之间有三个主要区别:

1. 基本类型只有值,而装箱基本类型具有与他们的值不同的同一性;
2. 基本类型只有功能完备的值,而每个装箱基本类型除了它对应基本类型的所有功能值之外,还有个非功能值:null;
3. 基本类型通常比装箱基本类型更节省时间和空间.

用到装箱基本类型的情况:

1. 作为集合中的元素,键和值,必须将装箱基本类型放在集合中;
2. 在参数化类型中,必须使用装箱基本类型作为类型参数,因为java不允许使用基本类型;
3. 进行反射的方法调用中,必须使用装箱基本类型.

dubbo(二)

一、dubbo的负载均衡策略

1.随机（random）

      随机选取一台服务器去运行
      不足：数量比较少的时候，负载均衡的效果不是很明显

2.轮询（roundrobin）

      第一次调用第一台，第二次调用第二台服务器，以此类推
      不足：假设有十台服务器其中第二台响应特别慢，调用的时候不会立即结束，当下一个服务进来的时候，上一个服务没有立即结束，服务器就不会认为是一个完整的服务，所以下一个服务就会停留在第二台服务器上去进行响应，由于第二台服务器响应依旧很慢，以此类推越积越多，都聚集到这个比较慢的服务器上去了。非常拥挤。

3.最小活跃数原则（leastactive）

      哪台服务器调用的最少，就去调用哪台服务器
      相对来说比较合理

4.一致性hash原则（consistenthash）

      同一个客户端始终调用同一台服务器

二、如何指定负载均衡策略？

      在实际应用中，默认是随机random的。指定的时候，在消费者一端，consumer.xml的配置文件中，dubbo:reference 中有一个属性（loadbalance），指定想要遵循的负载均衡策略

      注意：如果在服务端设置了负载均衡，客户端没有设置，则默认走服务端的设置；如果同时设置了负载均衡，则走客户端的负载均衡。

1.开启注解

2.服务端：在对应的包下的java文件中类前写注解@service

      注意：是引入的dubbo的service而不是spring的。

安装dubbo的监控中心以及管理控制台的安装

一、安装监控中心

1.创建安装目录

mkdir -p /usr/local/dubbo

2.解压到指定目录

tar zxvf dubbo-monitor-simple-2.5.3-assembly.tar.gz -C /usr/local/dubbo

3.修改配置文件

cd /usr/local/dubbo/dubbo-monitor-simple-2.5.3/conf

4.启动

cd /usr/local/dubbo/dubbo-monitor-simple-2.5.3/bin/
./start.sh


二、安装管理控制台

1.解压tomcat、清空tomcat/webapps/ROOT目录下的文件

tar zxvf apache-tomcat-7.0.68.tar.gz -C /usr/local/dubbo
rm /usr/local/dubbo/apache-tomcat-7.0.68/webapps/ROOT/* -rf

2.解压dubbo-admin.war至tomcat/webapps/ROOT目录

unzip dubbo-admin-2.5.4.war -d /usr/local/dubbo/apache-tomcat-7.0.68/webapps/ROOT

3.修改dubbo.properties文件

cd /usr/local/dubbo/apache-tomcat-7.0.68/webapps/ROOT/WEB-INF

4.启动tomcat，访问

cd /usr/local/dubbo/apache-tomcat-7.0.68/bin/startup.sh

dubbo(一)

一、dubbo

       分布式企业级分布服务框架。可以用来做服务的负载均衡，将客户端浏览器的访问均摊到服务器之中。将service由一变多。

二、背景

单一应用架构

       当网站流量很小时，只需一个应用，将所有功能都部署在一起，以减少部署节点和成本。此时，用于简化增删改查工作量的数据访问框架(ORM)是关键。

垂直应用架构

       当访问量逐渐增大，单一应用增加机器带来的加速度越来越小，将应用拆成互不相干的几个应用，以提升效率。此时，用于加速前端页面开发的Web框架(MVC)是关键

分布式服务架构

       当垂直应用越来越多，应用之间交互不可避免，将核心业务抽取出来，作为独立的服务，逐渐形成稳定的服务中心，使前端应用能更快速的响应多变的市场需求。此时，用于提高业务复用及整合的分布式服务框架(RPC)是关键。

流动计算架构

       当服务越来越多，容量的评估，小服务资源的浪费等问题逐渐显现，此时需增加一个调度中心基于访问压力实时管理集群容量，提高集群利用率。此时，用于提高机器利用率的资源调度和治理中心(SOA)是关键。
